# template

프로젝트의 전체적인 목적과 방향을 한 단락으로 요약합니다.  
예: 기존 시스템의 구조적 문제를 해결하기 위해 새로운 데이터 모델, 자동화 파이프라인, 협업 구조를 재정비했습니다.

## 1. 업무 개요

- 역할: 담당 역할 (예: 프론트엔드 / 풀스택 / 아키텍처 설계 등)  
- 기술 스택: 기술 스택 나열  
- 업무 목표:
  - 해결하려 했던 비즈니스/기술적 목표 1  
  - 해결하려 했던 비즈니스/기술적 목표 2  
  - 선택적으로 조직 또는 사용자 관점 목표 포함  



## 2. 문제와 해결

여러 개의 문제를 각각 독립된 섹션으로 구성해 설명합니다.


### 2-x. 문제 제목 (예: 문항 구조 하드코딩으로 인한 확장성 부족)

#### 🚧 문제

당시 시스템이 갖고 있던 문제를 명확하게 정의합니다.  
어떤 컨텍스트에서 발생했는지, 왜 중요한 문제인지, 어떤 영향을 주었는지 설명합니다.

#### 🧪 근거

문제를 뒷받침하는 실제 사례를 정리합니다.  
구조적 한계, 업무 흐름 상의 병목, 발생했던 오류들, 팀 간 충돌, 시간/비용 낭비 등  
정량·정성 근거를 모두 포함할 수 있습니다.

- 예: 문항 로직이 화면·스크립트 곳곳에 분산되어 유지보수가 어려웠음  
- 예: 대량 설문 생성 시 반복 작업 발생 → 오류 증가  
- 예: 역할 분리가 없어 연구팀·개발팀 간 커뮤니케이션 비용 증가

#### ✅ 해결

문제를 해결하기 위해 취한 접근 방식을 단계적으로 설명합니다.  
기술적 구조 변경, 모델링 방식 개선, 프로세스 변화, 자동화 도입 등을 포함할 수 있습니다.

- 해결 접근 1  
- 해결 접근 2  
- 해결 접근 3  
- 필요 시 Before/After 설명을 문장으로 추가해 비교를 명확히 합니다.

---

## 3. 성과

개선 후 나타난 효과를 정리합니다.

- 정량적 성과 예: 작업 시간 70% 감소, 오류율 40% 감소, 공수 40% 절감 등  
- 정성적 성과 예: 협업 구조 개선, 확장성 확보, 유지보수성 향상  
- 조직 또는 사용자 관점 효과  
- 장기적으로 얻은 이점  

---

## 4. 회고 (선택)

- 문제 해결 과정에서 얻은 인사이트  
- 다음 프로젝트에서 적용할 개선 아이디어  
- 조직/프로세스 측면에서 느낀 점
